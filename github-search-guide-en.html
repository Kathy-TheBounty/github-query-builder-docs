1. How it works

Keywords vs Qualifiers

Free keywords Qualifiers qualifier:value
Search across multiple fields simultaneously Target one specific field only
Support explicit AND, OR, NOT OR via repetition, NOT via -qualifier:val
"machine learning" → exact phrase language:python → specific field
💡 Golden rule: Explicit AND/OR/NOT only work between free keywords. For qualifiers, everything is implicit — space = AND, repetition = OR, dash = NOT.

2. Users Search

What keywords search

A keyword without qualifier searches simultaneously in: username, display name, public email (local part only), profile bio.

💡 The bio is the richest field for sourcing. Devs list their stack, title, and specialties there.

Available qualifiers

QualifierTargeted fieldExampleNotes
language: Primary language of public repos language:python OR implicit if repeated
location: Profile location field (declarative) location:Paris OR implicit if repeated
followers: Number of followers followers:>50 followers:20..200 Supporte ranges
repos: Number of public repos repos:>10 repos:5..50 Includes forks
created: Account creation date created:>2019-01-01 Format ISO YYYY-MM-DD
in:login Username (partial search) python in:login Keyword + in:
in:name Nom affiché Sophie in:name Keyword + in:
in:email Public email (local part) - before the @ dataiku in:email Not the domain
type: Account type type:user Excludes organizations

Location — the blind spot

The location: field is entirely declarative. A developer in Paris may write "Paris", "75", "Île-de-France", "Greater Paris Area" or nothing at all.

⚠️ The only way to cover an area exhaustively: run multiple queries in parallel with all variants — and aggregate the results.
type:user language:python location:Paris location:"Île-de-France" location:France location:75 followers:>20

Users query examples

"ML engineer" language:python location:Paris location:France followers:>20 repos:>5 NOT "junior" "tech lead" OR "engineering manager" location:France language:typescript repos:>5 "open to work" "python" location:Paris language:python

3. Repos Search

What keywords search by default

Without in:, a keyword searches name + description + topics. Not the README.

The in: qualifier

Repos → Users strategy

The real power: find relevant repos → extract their owners and contributors → enrich their profiles. This bypasses the key limitation of user search, which cannot filter by specific framework or library.

Combination examples

// Implicit AND between different qualifiers language:python location:France repos:>10 followers:>50 // Implicit OR — same qualifier repeated location:Paris location:Lyon location:Bordeaux location:Nantes // Combining keywords and qualifiers "machine learning engineer" location:Paris location:"Île-de-France" language:python followers:>20 // Exclusions location:France -language:php -language:java NOT "junior" NOT "intern"
⚠️ Not allowed: OR between different qualifiers (location:Paris OR language:python) and AND between same qualifiers (language:python AND language:js) do not work. Workaround: two separate queries.

5. Limits & Workarounds

Beyond 1,000 results — time-based pagination

When a query returns exactly 1,000 results, there are more. Split using created::

"python developer" location:France created:<2018-01-01 "python developer" location:France created:2018-01-01..2020-12-31 "python developer" location:France created:2021-01-01..2022-12-31 "python developer" location:France created:>2023-01-01

AND between languages — intersection technique

Impossible in one query. Simulate via intersection:

Query 1 : language:python location:France → list_A Query 2 : language:typescript location:France → list_B Intersection A ∩ B → profiles present in both lists
⚠️ This technique gives an approximation — a versatile developer may only appear in one list if their repos in the other language are a minority.

Exclude noise languages

machine learning language:python -language:"jupyter notebook" -language:html stars:>20

6. Cheat Sheet — Complete Reference

Keywords vs Qualifiers — the fundamental difference

Free keywords Qualifiers qualifier:value
What it is Terms typed without special syntax, with or without quotes qualifier:value syntax targeting a specific field
Where it searches — Users Username · display name · public email (local part) · bio One specific field only (e.g. language: → primary language)
Where it searches — Repos Name · description · topics (default) — changeable with in: One specific attribute only (e.g. stars: → star count)
AND Juxtapose: "backend" "python" Juxtapose different qualifiers: language:python location:France
OR Keyword OR keyword: "frontend" OR "front-end" (max 5 operators) Repeat the same qualifier: location:Paris location:Lyon
NOT / Exclude Keyword NOT keyword: "developer" NOT "junior" Prefix with -: -language:php
⛔ What doesn't work OR/NOT do not work on qualifiers Explicit OR/AND do not work between qualifiers

Logic — keywords (Users & Repos)

SyntaxBehaviorExample
Keyword Single word — searches the term in default fields developer
"expression exacte" Words must appear in this exact order "machine learning engineer"
"Keyword1" "Keyword2" Implicit AND — both expressions must be present "backend" "python"
"A" OR "B" Explicit OR — either one is enough. Max 5 OR/NOT operators per query. "frontend" OR "front-end" OR "ui developer"
"A" NOT "B" Explicit NOT — excludes B from all searched fields "developer" NOT "junior" NOT "intern"
Keyword in:field Restricts search to a specific field (users: login/name/email — repos: name/description/topics/readme) python in:login · LangChain in:readme
"A" OR "B" in:field OR between keywords + targeting a specific field "llm" OR "rag" in:name,description
⚠️ Reminder: Explicit OR and NOT only work between keywords. They do not apply to qualifiers.
